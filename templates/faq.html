Frequently Asked Questions

Q: How did you break down the project?
A: The project was broken down into five main phases, following key recommended steps:
  • Phase 1 - API Selection and Testing (1 week):
    • Researching APIs that meet criteria (REST, JSON, no quota)
    • Testing endpoints with Postman
    • Documenting API limitations and responses
    • Validating the choice with the supervisor
  • Phase 2 - Design and Wireframing (1 week):
    • Creating wireframes for all required pages
    • Designing the user interface according to technical constraints
    • Validating the site structure
  • Phase 3 - Development (2 weeks):
    • Implementing mandatory features (FT1 to FT4)
    • Developing the search and filtering system
    • Creating the pagination system
    • Setting up a favorites system with persistence
  • Phase 4 - Testing and Debugging (3 days):
    • Testing all functionalities
    • Checking error handling
    • Testing browser compatibility
    • Fixing identified bugs
  • Phase 5 - Documentation and Finalization (4 days):
    • Writing the README
    • Documenting routes
    • Creating the about page
    • Preparing the final deliverable
  • Phase 6 - Issue Resolution and Optimization (1 week):
    • Fixing major UI bugs
    • Resolving API performance issues
    • Improving error handling
    • Optimizing caching and response times
    • Fixing browser compatibility issues
    • Enhancing data validation
    • Resolving favorite persistence issues
    • Optimizing search and filtering queries
Q: How did you manage your time?
A: Time management was structured around several key aspects:
  • Planning:
    • Weekly progress checkpoints
    • 3-day buffer for unexpected issues
    • Setting priorities at the beginning of the week
    • Alternating between work sessions and breaks
  • Prioritization:
    • Initial focus on mandatory features
    • Development based on technical dependencies
    • Continuous testing to prevent bug accumulation
    • Bonus features only after essentials were completed
  • Daily Organization:
    • 2 to 4 hours of development per day
    • Testing and debugging in the afternoon
    • Documentation at the end of the day
    • Daily code review
  • Risk Management:
    • Identifying critical points from the start
    • Backup solutions for complex features
    • Monitoring API limitations
    • Regular progress tracking
